# 中文年报 RAG 项目总结

## 当前项目定位

当前仓库已经从早期实验状态收敛为一条面向中文上市公司年报问答的主线工程，核心能力包括：

- 使用 `MinerU` 解析中文年报 PDF
- 读取 `content_list.json` 并重建页级内容
- 使用 `bge-m3` 构建向量检索
- 使用 `BM25 + 向量检索 + RRF` 做混合召回
- 使用 `Cross-Encoder` 或 LLM 做精排
- 使用统一 Prompt 和统一 Schema 完成结构化问答

当前项目更强调：

- 中文长文档解析稳定性
- 检索链路可解释性
- 结构化问答的一致性
- 页码引用与结果可追溯性

## 为什么统一使用 MinerU

在中文年报场景中，解析器的稳定性比单次效果更重要。当前项目统一使用 `MinerU`，主要原因是：

- 对中文文档和复杂版式的适配更稳定
- 便于稳定输出页级结构，适合后续 RAG 使用
- 可以较好保留页码、表格和文本块边界

因此，当前项目不再保留早期多解析器并行实验的复杂分支，而是统一收敛到 `MinerU` 主线。

## 数据处理中间格式

项目不会直接拿 PDF 或 Markdown 做后续检索，而是采用以下中间流程：

1. 使用 `MinerU` 解析 PDF
2. 读取 `*_content_list.json`
3. 重建为项目内部统一 JSON 格式
4. 基于统一 JSON 做切块、建库和检索

这样做的价值在于：

- 保留页码信息，便于引用与溯源
- 让解析器输出与检索链路解耦
- 为父文档召回和结构化问答提供稳定边界

## 检索与问答设计

### 混合检索

当前主链采用：

- HNSW 向量检索
- BM25 中文关键词检索
- RRF 粗排融合
- 父文档召回
- Cross-Encoder 或 LLM 精排

### 统一问答

当前主链已收敛为统一 Prompt 与统一 Schema，支持：

- `name`
- `number`
- `boolean`
- `names`
- `comparative`

同时已经补充：

- 题型提示 `question_kind_hint`
- 答案形态提示 `answer_shape_hint`
- 数值题单值输出约束
- `answer_unit` 与 `unit_basis` 结构化字段

### 程序化增强

为减少静默错答，当前项目还引入了程序化增强：

- 数值题标准化与单位校验
- 字段题格式校验
- 比较题程序化比较
- 统一答案校验层

## 当前建议的主配置

如果以稳定性为优先，当前更推荐：

- 纯 RAG 基线：`qwen_max_ce_top5_rag_only`
- 本地向量模型：`bge-m3`
- 本地交叉编码器：`bge-reranker-v2-m3`
- 主回答模型：按 `.env` 中 `QWEN_CHAT_MODEL` 配置

## 当前项目状态

当前项目已经具备：

- 交互问答能力
- 批量问答能力
- 检索与精排配置切换能力
- 结构化输出与调试结果落盘能力

整体状态可以概括为：

- 已经成型
- 可以做内部演示和小范围试用
- 仍需继续做一致性优化与收口

## 后续优先方向

后续如果继续迭代，建议优先：

1. 继续提升布尔题和比较题召回稳定性
2. 统一交互链与批处理链的高价值检索策略
3. 逐步清理历史残留函数和说明文档
4. 完善最终结果校验与失败回退机制
