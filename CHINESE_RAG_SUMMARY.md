# 中文年报 RAG 改造总结

## 当前项目定位

当前仓库已经不再保留原始竞赛项目的大量历史分支，而是收口为一条面向中文企业年报的主线：

- `MinerU` 解析中文年报 PDF
- `content_list.json` 重建为按页文本
- 中文切块
- `bge-m3` 编码
- HNSW 向量检索 + BM25 关键词检索
- 父文档召回
- LLM / Cross-Encoder 两条精排链路
- 规则抽取 + RAG 兜底问答

## 为什么替换 docling

在实际中文年报数据上，`docling` 的核心问题不是“效果一般”，而是稳定性不够：

- 部分文档直接转换失败
- 中文 OCR 和复杂版式场景不稳定
- 不利于稳定产出后续 RAG 需要的页级结构

因此当前项目统一使用 `MinerU` 作为解析入口。

## 中间格式设计

项目不会直接拿 PDF 或 markdown 做后续检索，而是：

1. 使用 `MinerU` 解析 PDF
2. 读取 `*_content_list.json`
3. 按页重建为项目内部 JSON
4. 再做中文切块和建库

这样做的价值是：

- 保留页码，便于引用和溯源
- 为父文档召回提供稳定边界
- 把解析器输出和检索链路解耦

## 检索与问答设计

当前系统采用两层策略：

### 规则抽取优先

优先处理结构化强的问题，例如：

- 法定代表人
- 公司网址
- 股票代码
- 营业收入
- 归母净利润
- 研发投入
- 基本每股收益

规则路径抽不到，再回退到 RAG。

### 混合检索兜底

RAG 主链当前采用：

- HNSW 向量检索
- BM25 中文关键词检索
- RRF 粗排融合
- 父文档召回
- LLM 或 Cross-Encoder 精排

其中：

- `qwen_max_llm_top5`：LLM 精排 top5
- `qwen_max_ce_top5`：粗排 top30，交叉编码器精排 top5

## 当前工程重点

这个项目当前更强调：

- 中文长文档解析稳定性
- 检索链路可解释性
- 结构化问题的低误答率
- 页码引用和结果可追溯

而不是保留原始仓库里的所有实验分支。

## 后续建议

如果继续迭代，优先级建议是：

1. 扩章节 / 标题 metadata
2. 提高跨页表格处理能力
3. 扩大中文评测集并分题型评估
4. 继续细化布尔题和比较题策略
5. 再考虑更大规模的索引和服务化能力
